# CNS: A Finality-Oriented Consensus Network Simulator

CNS is a Java-based, event-driven, object-oriented simulation framework for evaluating blockchain networks and consensus protocols, with a particular focus on **transaction finality**. It is distributed under the LGPL license and provides a common set of classes that can be reused or specialized to simulate arbitrary consensus protocols.

Key features:

- Transaction-level simulation.
- Abstract modelling of network topologies and end-to-end transmission speeds.
- Extensible event and node-behavior object models.
- Externally loaded or internally generated samples (workloads, nodes, topologies, and so on).
- Customizable reporting.
- Built-in support for finality analysis through controlled seed management.

## Architectural Overview

Following standard event-driven simulator design practice, CNS is organized around a queue of future events ordered by their expected time of occurrence. The heart of the simulator is a main loop in which, on every iteration:

1. The next event is popped from the queue.
2. The event's type and payload are used to update the state of the simulator's components.
3. New events stemming from that occurrence are scheduled into the queue.

For example, a `NewTransactionArrival` event — representing a `Transaction` object entering the system at a specific node — triggers the scheduling of `N-1` `TransactionPropagation` events on the remaining `N-1` nodes of the network; the exact arrival times depend on the end-to-end bandwidth between nodes.

A global variable tracks **simulated time**, which jumps from one instant to the next according to the event currently being processed. The main loop terminates when the queue is empty or when a designated termination event is popped from it.

## Nodes and Behaviors

The logic of a consensus protocol is implemented within the abstract `Node` type by specifying how a `Node` instance reacts to events. The `Node` type itself contains only limited, protocol-neutral functionality, restricted to capturing events through `event_[X]` methods, where `[X]` is the event the node is reacting to.

Protocol-specific functionality is delegated from the `Node` to a companion `[Y]Behavior` object, where `[Y]` characterizes the behavior in question (for example, honest, malicious, impaired, or implementing a specific attack). Delegation follows the **Strategy** pattern, so that a node can switch behaviors mid-simulation — for instance, becoming malicious from honest at a given point in a consensus process.

An extensible static `Reporter` class is used by the various components of the simulator to persistently save log-worthy events.

## Workloads, Networks, and Samplers

Network, node, and workload instances can either be created externally and loaded at the start of a simulation, or generated by the simulator itself based on configurable sampling parameters.

- **Workloads** are sets of `Transaction` objects, each carrying a size and a fee value, arriving at specific nodes at given times. CNS's default built-in workload generator supports Poisson arrivals together with normally distributed sizes and values; both distributions are parameterizable.
- **Node and network characteristics** — such as mining power, energy costs, and end-to-end bandwidth — can also be loaded from files or generated using built-in samplers, which by default use the normal distribution.
- **Ledger structure.** Each `Node` holds a reference to a `Structure` object that represents the network's ledger. The `Structure` type is meant to be specialized with the concrete implementation required by the protocol in question — for example, a linked list of blocks, or a DAG.
- **Samplers.** Several `Sampler` types are used for random number generation, one per major component of the simulator (transactions, network, nodes). The most important is the node sampler, which generates initial random node characteristics (when these are not loaded from a file) and is then used continuously to produce stochastic outcomes related to node behavior — for example, the time to the next validation in Bitcoin, or tip selection in DAG-based protocols.

## Extending CNS

The above functionality is bundled in an `Engine` package and exposed as an object-oriented framework. To implement a specific protocol, developers need to:

1. Implement a specialization of the `Node` type (and factories thereof).
2. Specify the node's behaviors — that is, how it responds to events.
3. Implement the corresponding `Structure` type for the protocol's ledger.
4. Provide a main class that creates and assembles the individual objects from a configuration file.
5. Optionally, specialize the `Reporter` class to include protocol-specific reports.

## Finality Analysis

A core use case of CNS is the production of `|K|` copies of the state of the blockchain network at a time of interest, each of which is then evolved independently using a different random seed. CNS accomplishes this through customized seed management.

When configured for finality analysis, CNS runs `|K|` simulations using the **same** seed from time `0` up to a chosen time `t_0`. From that point forward, the samplers are re-seeded with a value derived from the experiment's ID. As a result, all simulations share a common build-up to the state in which the sampled transaction arrives, but evolve independently thereafter — yielding the variability needed for finality analysis while keeping the simulations fully reproducible through seed reconstruction.

This approach also simplifies use of the tool: experimenters do not need to construct, save, and reload concrete initial conditions in order to share a common starting state across runs.

## Reporting Beliefs

To capture finality information, CNS defines a special `BeliefReport` event. When this event occurs at a `Node`, it triggers a routine that, for a predefined sample of transactions, registers with the `Reporter` the degree to which the node believes each sampled transaction to be valid, according to a user-defined threshold `d`.

A sequence of such `BeliefReport` events, equidistantly spaced according to user configuration, is scheduled in the event queue at the start of the simulation. At the end of the run, the `Reporter` outputs a **belief table** that lists, for each combination of simulation ID, sampling time, and node ID, the corresponding degree of belief in the sampled transaction.

External data analysis scripts can then process this belief table to compute finality estimates and to produce aggregations and visualizations such as belief-evolution graphs.
